# 00 — Product Brief (EDM v2)

## 0.1 Πρόβλημα που λύνουμε
Στην πράξη, τα δεδομένα προϊόντων/τιμών/χαρακτηριστικών έρχονται από προμηθευτές σε διαφορετικές μορφές:

- PDF τιμολόγια/προσφορές με πίνακες
- XML (όταν υπάρχει)
- Excel λίστες προϊόντων
- φωτογραφίες/σαρωμένα έγγραφα
- καταλόγους PDF
- website προμηθευτή

Τα δεδομένα είναι συχνά **ασυνεπή**, με μικρές παραλλαγές κωδικών/ονομάτων, και απαιτούν **χειροκίνητη δουλειά**:

- να βρεθεί ο σωστός κωδικός προϊόντος,
- να γίνει mapping/ταυτοποίηση,
- να ενημερωθούν τιμές/στοιχεία,
- να εμπλουτιστούν τεχνικά χαρακτηριστικά.

Το EDM v2 στοχεύει να μειώσει δραστικά τον χρόνο αυτό, με **κανόνες + AI υποβοήθηση + ουρά ελέγχου**.

## 0.2 Ποιοι είναι οι χρήστες
- **Data operator / back office**: ανεβάζει αρχεία, διορθώνει, εγκρίνει.
- **Procurement/owner**: θέλει εικόνα για τιμές/ιστορικό.
- **Admin**: ορίζει suppliers, κανόνες, δικαιώματα.

## 0.3 Τι είναι “προϊόν” στο EDM
Το “προϊόν” είναι μια οντότητα στην κεντρική βάση του EDM.

- Έχει **stable internal id** (UUID) και εσωτερικό κωδικό (αν υπάρχει).
- Συνδέεται με προμηθευτές μέσω **ProductSupplierLink**.
- Οι κωδικοί προμηθευτή/manufacturer είναι **primary signals** για matching.

> Κρίσιμο: Η λογική matching **δεν** βασίζεται σε Pylon codes (όπως συμφωνήθηκε).

## 0.4 Τι σημαίνει “matching”
Matching = απόφαση ότι μια γραμμή παραστατικού αντιστοιχεί:

- είτε σε ήδη υπάρχον προϊόν,
- είτε σε νέο προϊόν (προσωρινό/provisional) που πρέπει να εμπλουτιστεί.

Matching δεν είναι “μαύρο κουτί”. Πρέπει να είναι εξηγήσιμο με:

- τον κανόνα που εφαρμόστηκε (deterministic), ή
- το score / rationale και τα candidates (AI assist),
- το τελικό review decision.

## 0.5 Τι σημαίνει “enrichment”
Enrichment = συμπλήρωση/βελτίωση των δεδομένων προϊόντος (π.χ. τεχνικά χαρακτηριστικά, φωτογραφίες, datasheets, categories).

Η σειρά πηγών είναι **αυστηρή**:

1) XML/Excel supplier lists (δομημένα)
2) catalog/product lists που ανεβάζει ο χρήστης
3) manual συμπλήρωση
4) web scraping (τελευταίο)

## 0.6 “Supplier Agreement” (συμφωνητικά)
Τα συμφωνητικά:

- ανεβαίνουν,
- αρχειοθετούνται,
- είναι searchable (π.χ. RAG / semantic search),

αλλά **δεν** χρησιμοποιούνται για αυτόματο compliance check με τιμολόγια στο scope του v2, εκτός αν ζητηθεί ρητά.

## 0.7 Πρώτο production-like pilot
- Supplier: **Poimenidis**
- Inputs: invoices με “σωστή δομή”, XML, φωτογραφίες, website search.

Στόχος pilot:
- να περάσουμε end-to-end flow με υψηλή επιτυχία και χαμηλό manual χρόνο.
